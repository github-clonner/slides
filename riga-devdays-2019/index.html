Traefik Core Concepts

Let’s dive a bit inside the core concepts. So we know what problem we want to solve. Being able to easily route the request from the outside to inside without to write as much configuration as we can. We only want you pass time on important things and not time to configure the reverse proxy.

Let’s Simplify

But we are here to understand how this working and how we help you to solve the problem. Just open the box and jump inside traefik. For that I will take a singe example. One domain name for one website runing inside one docker container and one traefik instance. Auto discovery line, this means that traefik will auto discovers his configuration, and will be able to dynamically configure itself based on what it can find inside docker.

Providers

That the core concept. Step one, we need to tell traefik what are the provider of application that you need to route the request inside. So you need to connect traefik to docker or kubernetes. And once it has that knowledge which means credentails and an access. it’s ready to fetch informations.

Entrypoints

You still have one thing at least to configure. You have to tell traefik which port it’s going to listen on. because it’s depdends on your ocnfiguration and cannot be guessed by fetching the prodvider. So that’s the only static information that traefik requires inside the config file or command line flag. This is the entrypoint you can have multiple. One entrypoint say I have one port, whatever domain, whatever content of the request. And some associated propreties. Do I want TLS temrination on that entrypoint ? Do i want pure HTTP

Backends

The second concept is here only for your knowledge because you don’t have to care about that. The are the backends. Backend mpas a running applicaiton in your backend system. These object are generated and maintain, updated by traefik without anything for you to do. It’s extracting by fetching the provider. For example a docker container has an IP and an exposed port. Traefik watch the docker API and when the container, start, restart or rescheduled somewhere else it will get the IP and port and will materialize that as a backend object. So you can send some request to that.

Frontends

Finally the thrid concept how we connect request entering the entrypoint and a backend ? There are the frontends. Tehre are object that say ok I receive the request, let’s analyze this request, eventually change it and route that request to the backend based on a rule. The rule could be host route, path route, header route method route. you can also change the request, rewrite the path, change some header, add some, removes some. That part is almost automated. you don’t have to tell traefik these rules. Traefik will fetch or guess these rules from the provider. I need your attention now, because it means that the developer the person deploying the backend application are now providing the information saying okay we have Jenkins, let’s deploy jenkins as a backend. I want to have jenkins to be reachable to that hostname, on that path with that authentication. These are meta information that are deployed with the service so when traefik only fetch these information and infer and create the rules for you.

In Practice

In practive you have a incomming request on backoffice.domain.com host. It’s come on entrypoint http port 80, traefik will detect the host and this host match with a frontend rule, so this request is going trougth the backend object and it’s forward to the correct container.

HTTPS for Everyone with Let’s Encrypt

  • TLS, DNS and HTTP challenges supported

ACME is a protocol to request certificate to let’s encrypt or toher certificate provider. That means you have a complete protocol to say hello i would want a TLS certificate for that domain. You request that to a public provider. Let’s encrypt is free. There are steps to follow before the provider delevers you a certificate. Traefik will be able to manage these step for you, generate and renew the certificates for you without for you to do anything. Challenge, when you request a certificate you have to prove that you are the person you seek to be. Let’s encrypt or any acme provider will challenge you. UID on http page, will be deleted after. TLS Let’s encrypt try to connet in https to your webserver and inside the certificate during ht handshake the is the token. DNs will create TXT record on your DNS provider and LE will check this TXT record.

The Herd

You came to the wrong neighbour

What if traefik as a single binary goes down? You can not manage incomming request time to traefik to be restarted and being in the shape.

Traefik comes in Herd

That’s why we made it distributed. It is an enterprise edition

This is how it works. Support

The goad is to provide high availability, so if one of your nodes goes down the traffic is distributed to the other nodes. When either you are on metal, swarm or kubernetes.

It’s also provide you safety, you need credential to configure traefik for let’s encrypt, your backend provider. You don’t want these credentials to be on the place where you have external requests incomming. So we split in two components, - Control nodes responsible of configuration management to put on the private part of your cluster. - Data nodes that takes care of forwarding request to your backend.

And finally traefikee provide you sclability nodes responsibe to forwarding the request can scale time to under the burst and scale down to consume too much resources on your cluster.

Quick Glance

[entrypoints]
    [entrypoints.web-secure]
      address = ":443"
[http]
   [http.routers.to-service-1]
      rule = "Host(`demo.containous.cloud`)"
      service = "service-1"
      [http.routers.to-service-1.tls] # terminates the tls connection at HTTP
[tcp]
   [tcp.routers.to-service-2]
      rule = "HostSNI(`demo.containous.cloud`)"
      service = "service-2"
      [tcp.routers.to-service-2.tls] # terminates the tls connection at TCP
[tcp.routers.to-service-3]
      rule = "HostSNI(`demo.containous.cloud`)"
      service = "service-3"
      [tcp.routers.to-service-3.tls]
         passthrough = true # sends encrypted data "as is" to service-3
